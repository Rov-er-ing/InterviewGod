# Product Requirements Document (PRD)

## Signal Detection System вҖ” Mass Hiring / Scale-Up Signal Detector

**For: Vikaas.ai Platform | Product: InterviewGod**
**Version: 1.0 | Date: April 17, 2026**

---

## 1. Overview

### 1.1 Purpose

This PRD defines the requirements for building a local, serverless-style signal detection system in Python that identifies mass hiring and scale-up signals from public data sources. The system will feed structured signals into the Vikaas.ai platform to power contextual outreach for InterviewGod.

### 1.2 Problem Statement

Sales and outreach teams need timely intelligence about companies entering large hiring phases. Manually tracking news, press releases, and job boards is slow and unscalable. An automated pipeline that ingests public data, applies rule-based detection, and outputs machine-readable signals will dramatically reduce this manual overhead.

### 1.3 Goals

- Ingest public news and article data for a configurable list of target companies
- Detect hiring-related signals using keyword and numeric pattern matching
- Score each signal by confidence
- Output structured JSON records consumable by downstream systems
- Remain entirely free, local, and dependency-light

### 1.4 Non-Goals

- No LLM or AI API usage
- No cloud deployment
- No paid data sources
- No real-time streaming (batch/scheduled runs only)

## 3. Module Specifications

### 3.1 Configuration (`config.yaml`)

The config file is the single source of truth for runtime behavior. It must support:

| Parameter | Type | Description |
|---|---|---|
| `companies_file` | string | Path to companies input JSON |
| `output_dir` | string | Where to write signal files |
| `output_format` | enum: `json`, `sqlite` | Output storage format |
| `rss_sources` | list of URL templates | RSS feed patterns with `{query}` placeholder |
| `score_threshold` | int (0вҖ“100) | Minimum score to emit a signal |
| `fetch_timeout_seconds` | int | HTTP request timeout |
| `max_articles_per_company` | int | Cap articles fetched per company |
| `dedup_window_hours` | int | Suppress re-emitting same article within N hours |

### 3.2 Input Data (`data/companies.json`)

```json
[
  {
    "name": "Infosys",
    "aliases": ["Infosys Ltd", "Infosys Technologies"],
    "domain": "infosys.com",
    "industry": "IT Services"
  },
  {
    "name": "Zomato",
    "aliases": ["Zomato Ltd"],
    "domain": "zomato.com",
    "industry": "Food Tech"
  }
]
```

**Fields:**

| Field | Required | Description |
|---|---|---|
| `name` | Yes | Canonical company name used in queries |
| `aliases` | No | Alternate names for matching in article text |
| `domain` | No | Used for source URL filtering |
| `industry` | No | For downstream enrichment/filtering |

### 3.3 Fetcher Module (`signals/mass_hiring/fetcher.py`)

**Responsibility:** Retrieve raw article data from public sources for each target company.

**Data Sources (in priority order):**

| Source | URL Pattern | Notes |
|---|---|---|
| Google News RSS | `https://news.google.com/rss/search?q={company}+hiring` | Primary source |
| Google News RSS (variant) | `https://news.google.com/rss/search?q={company}+expansion+jobs` | Secondary query |
| Bing News RSS | `https://www.bing.com/news/search?q={company}+hiring&format=RSS` | Fallback |
| Feedspot / aggregators | Configurable | Optional fallback |

**Fetch Logic:**

1. For each company, construct N query variants (hiring, expansion, campus drive, etc.)
2. Fetch each RSS feed with timeout and retry (max 3 attempts, exponential backoff)
3. Parse XML with `feedparser` or `xml.etree.ElementTree`
4. Extract per-article: `title`, `link`, `published`, `summary`
5. For articles where the summary is thin (<100 chars), attempt full-page fetch with `requests` + `BeautifulSoup`
6. Deduplicate by URL within the dedup window
7. Return a list of raw article dicts

**Error Handling:**

- Network timeout вҶ’ log warning, skip source, continue
- HTTP 4xx/5xx вҶ’ log error with status code, skip
- Malformed XML вҶ’ log parse error, skip feed
- Rate limiting (429) вҶ’ back off 60 seconds, retry once

**Output per article:**

```python
{
  "company_name": str,
  "title": str,
  "url": str,
  "published_at": str,   # ISO 8601
  "raw_text": str        # title + summary + body (if fetched)
}
```

---

### 3.4 Parser Module (`signals/mass_hiring/parser.py`)

**Responsibility:** Extract structured signals from raw article text.

**Parsing Steps:**

1. **Text normalization** вҖ” lowercase, strip HTML tags, collapse whitespace
2. **Company mention validation** вҖ” confirm company name or alias appears in text; discard if not
3. **Keyword extraction** вҖ” scan for hiring signal keywords (see В§3.5)
4. **Numeric extraction** вҖ” use regex to find hiring volume numbers
5. **Context window** вҖ” for each keyword match, extract Вұ30 words of surrounding context
6. **Entity tagging** вҖ” tag matches as `keyword_match` or `numeric_match`

**Numeric Pattern Regex (examples):**

```
\b(\d{1,3}(?:,\d{3})*)\s*\+?\s*(engineers?|developers?|employees?|hires?|jobs?|positions?|roles?|freshers?|candidates?)\b
\b(hiring|recruit)\s+(\d[\d,]*)\s*\+?
\b(\d+)k\s*(hires?|jobs?|employees?)
```

**Output per article:**

```python
{
  "company_name": str,
  "url": str,
  "published_at": str,
  "matched_keywords": list[str],
  "numeric_signals": list[{"value": int, "unit": str, "context": str}],
  "keyword_contexts": list[str],
  "raw_text_length": int
}
```

### 3.5 Keyword Definitions (`signals/mass_hiring/keywords.py`)

Keywords organized in tiers by signal strength:

**Tier 1 вҖ” Strong Hiring Signals (high weight):**
`mass hiring`, `large scale hiring`, `hiring spree`, `bulk recruitment`, `mega recruitment`, `aggressive hiring`, `10000 hires`, `1000 engineers`

**Tier 2 вҖ” Moderate Signals (medium weight):**
`hiring`, `recruiting`, `recruitment drive`, `campus drive`, `lateral hiring`, `fresher hiring`, `expansion`, `scaling up`, `new office`, `headcount`, `workforce expansion`, `job openings`

**Tier 3 вҖ” Weak / Contextual Signals (low weight):**
`talent acquisition`, `onboarding`, `new roles`, `open positions`, `we are growing`, `join our team`, `career opportunities`

**Negative Signals (reduce score):**
`layoffs`, `downsizing`, `retrenchment`, `job cuts`, `reduction in force`, `freeze hiring`, `hiring freeze`, `restructuring`

---

### 3.6 Scorer Module (`signals/mass_hiring/scorer.py`)

**Responsibility:** Assign a 0вҖ“100 confidence score to each parsed article signal.

**Scoring Formula:**

```
base_score = 0

# Keyword tier contributions
base_score += count(tier1_keywords) Г— 25    (cap at 50)
base_score += count(tier2_keywords) Г— 10    (cap at 30)
base_score += count(tier3_keywords) Г— 5     (cap at 10)

# Numeric signal bonus
if any numeric_signal with value >= 10000: base_score += 20
elif any numeric_signal with value >= 1000: base_score += 15
elif any numeric_signal with value >= 500:  base_score += 10
elif any numeric_signal with value >= 100:  base_score += 5

# Source credibility bonus
if source domain in trusted_domains list: base_score += 10

# Recency bonus
if published within 24h: base_score += 10
elif published within 7d: base_score += 5

# Negative signal penalty
base_score -= count(negative_keywords) Г— 15

final_score = clamp(base_score, 0, 100)
```

**Score Bands:**

| Range | Label | Meaning |
|---|---|---|
| 80вҖ“100 | `high` | Strong hiring signal, act immediately |
| 50вҖ“79 | `medium` | Likely signal, worth reviewing |
| 20вҖ“49 | `low` | Weak signal, monitor |
| 0вҖ“19 | `noise` | Filtered out (not emitted) |

**Reason Generation:**

The scorer must produce a human-readable `reason` string by combining the top matched signals. Example:

> "Large scale hiring announcement detected: 2 Tier-1 keywords matched ('mass hiring', 'bulk recruitment'), numeric signal of 5000 engineers found, published 3 hours ago."

### 3.8 Orchestrator (`signals/orchestrator.py`)

**Responsibility:** Wire all modules together and manage execution flow.

**Execution Flow:**

```
1. Load config.yaml
2. Load companies.json
3. For each company:
   a. fetcher.fetch(company) вҶ’ raw_articles[]
   b. parser.parse(raw_articles) вҶ’ parsed_signals[]
   c. scorer.score(parsed_signals) вҶ’ scored_signals[]
   d. Filter by score_threshold
   e. storage.write(scored_signals)
4. Log summary: N companies processed, M signals emitted
```

**Concurrency:** Use `concurrent.futures.ThreadPoolExecutor` with configurable `max_workers` (default: 4) to parallelize company-level fetch jobs.

## 7. Assumptions and Limitations

| # | Assumption / Limitation |
|---|---|
| 1 | Google News RSS may throttle aggressive scraping; the system applies delays and caps requests per run |
| 2 | Article summaries in RSS may be truncated; full-page fetch is attempted but not guaranteed |
| 3 | Numeric extraction is regex-based and may miss non-standard formats ("five thousand hires") |
| 4 | Score thresholds are heuristic and will need tuning against real data |
| 5 | No authentication is used; paywalled articles are skipped |
| 6 | The system does not deduplicate across runs beyond the configurable time window |
| 7 | Company name matching is string-based; subsidiary or brand names may be missed without aliases |
| 8 | RSS feed availability is subject to Google/Bing ToS changes |
